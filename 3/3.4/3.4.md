## **Caso de uso #1: Registrar Proveedor**

| **ID**                     | R-401 |
|-----------------------------|-------|
| **Actor(es)**               | Jefe de Abastecimiento |
| **Objetivo**                | Permite registrar un nuevo proveedor y los productos que ofrece, incluyendo datos como razón social, RUC, contacto, condiciones de pago, tiempo de entrega y precio unitario por artículo. |
| **Precondiciones**          | El usuario de abastecimiento debe estar logueado en el sistema.|
| **Disparador o evento inicial**              | El usuario ingresa al módulo de Abastecimiento y desea registrar un proveedor nuevo. |
| **Flujo Principal**         |  1. El usuario accede a la vista de Abastecimiento.<br>2. Selecciona el menú **Proveedores**.<br>3. Selecciona **"Registrar Proveedor"**.<br>4. Ingresa los datos principales del proveedor.<br>5. Registra los productos que ofrece y sus precios unitarios.<br>6. Confirma los datos para que sean ingresados.|
| **Postcondiciones**         | Existencia de una nueva entidad de Proveedor.|
| **Excepciones**             | 1. "Proveedor ya Existe" Cuando el proveedor ya se encontraba registrado.<br> 2. "Datos no Validos" Cuando se introducen datos con un valor no aceptado.<br>3. "Campo obligatorio vacío" Cuando se deja sin completar algún campo requerido para el registro.|
| **Frecuencia de Uso**       | Poco frecuente, cuando hay un nuevo proveedor o producto que ofrece |

---

## **Caso de uso #2: Actualizar datos del Proveedor**

| **ID**                     | R-402 |
|-----------------------------|-------|
| **Actor(es)**               | Jefe de Abastecimiento |
| **Objetivo**                | Permite modificar los datos registrados de un proveedor, incluyendo información de contacto, condiciones de pago o actualizar su catálogo de productos. |
| **Precondiciones**          | El usuario de abastecimiento debe estar logueado en el sistema, y el proveedor debe estar previamente registrado. |
| **Disparador  o evento inicial**              | Cambio en el valor de un atributo del Proveedor en la realidad.|
| **Flujo Principal**         | 1. El usuario accede a la vista de Abastecimiento.<br> 2. Selecciona el menú **Proveedores**.<br>3. Selecciona el proveedor a actualizar.<br>4. Modifica los campos necesarios del Proveedor.<br>5. Confirma los datos para que sean actualizados.<br>|
| **Postcondiciones**         | Cambio de datos del proveedor.|
| **Excepciones**             | 1. "RUC duplicado" Cuando el RUC ingresado ya se encuentra registrado en el sistema.<br>2. "Datos no válidos" Cuando se introducen datos con un valor no aceptado|
| **Frecuencia de Uso**       | Frecuente, cada vez que un proveedor cambia condiciones , contacto o precios de artículos. |

---

## **Caso de uso #3: Registrar Producto Nuevo**

| **ID**                     | R-403 |
|-----------------------------|-------|
| **Actor(es)**               | Jefe de Abastecimiento |
| **Objetivo**                | Permite Registrar un nuevo producto, con sus atributos como nombre, categoría, unidad de medida, proveedor asociado, precio de referencia. |
| **Precondiciones**          | El usuario debe estar logueado en el sistema. |
| **Disparador o evento inicial**              | El usuario ingresa al módulo de Abastecimiento y desea registrar un producto nuevo. |
| **Flujo Principal**         | 1. El usuario accede a la vista de Abastecimiento.<br> 2. Selecciona el menú **Productos**.<br>3. Selecciona **"Registrar Producto"**.<br>4. Ingresa los datos básicos del producto.<br>5. Confirma los datos para que sean ingresados. |
| **Postcondiciones**         | Existencia de una nueva entidad de Producto.|
| **Excepciones**             |1. "Datos no válidos" Cuando se introducen datos con un valor no aceptado.<br> 2. "Campo obligatorio vacío" Cuando se deja sin completar algún campo requerido para el registro.|
| **Frecuencia de Uso**       | Poco frecuente, cada vez que se agrega un nuevo artículo en la ferretería. |

---

## **Caso de uso #4: Atender Solicitud de Abastecimiento**

| **ID**                     | R-404 |
|-----------------------------|-------|
| **Actor(es)**               | Jefe de Abastecimiento |
| **Objetivo**                | Permite revisar y aceptar las solicitudes de abastecimiento generadas por el área de almacén o ventas.|
| **Precondiciones**          | El usuario debe estar logueado en el sistema y debe haber solicitudes de abastecimiento pendientes.|
| **Disparador o evento inicial**              | El usuario ingresa al módulo de Abastecimiento y desea atender una solicitud de abastecimiento. |
| **Flujo Principal**         |  1. El usuario accede a la vista de Solicitudes. <br> 2. Selecciona el menú **Atender Solucitud de abastecimiento**. <br>2. Selecciona una solicitud pendiente.<br>3. Visualiza los detalles de la solicitud .<br>4. “Rechaza” o “Acepta” la solicitud de abastecimiento.|
| **Postcondiciones**         | La solicitud de abastecimiento cambia su estado a **Aceptada** o **Rechazada** |
| **Frecuencia de Uso**       | Frecuente, según rotación de inventario o pedido para una venta. |

---

## **Caso de uso #5: Gestionar Solicitud de Abastecimiento**

| **ID**                     | R-405 |
|-----------------------------|-------|
| **Actor(es)**               | Jefe de Abastecimiento |
| **Objetivo**                | Permite organizar y priorizar las solicitudes aceptadas para definir qué productos se requieren con urgencia.|
| **Precondiciones**          | Solicitudes previamente aceptadas (R-404). |
| **Disparador o evento inicial**              | El usuario ingresa al módulo de Abastecimiento y desea gestionar una solicitud de abastecimiento.|
| **Flujo Principal**         |1. El usuario accede a la vista de Solicitudes. <br> 2. Selecciona el menú **Gestionar Solucitud de abastecimiento**.<br> 3. Visualiza todas las solicitudes.<br>4. Revisa cantidades y evalua urgencias.<br>5. Clasifica las solicitudes por prioridad<br> 6. Envía las solicitudes priorizadas al proceso de cotización.|
| **Postcondiciones**         | Solicitudes listas y priorizadas para cotizar. |
| **Frecuencia de Uso**       | Frecuente, según demanda. |

---

## **Caso de uso #6: Registrar Cotizaciones Recibidas**

| **ID**                     | R-406 |
|-----------------------------|-------|
| **Actor(es)**               | Jefe de Abastecimiento |
| **Objetivo**                | Permite registrar y asociar las cotizaciones enviadas por los proveedores a las solicitudes de abastecimiento Aceptadas.|
| **Precondiciones**          | Deben existir solicitudes de abastecimiento previamente aceptadas (R-405).|
| **Disparador o evento inicial**              |El usuario ingresa al módulo de Abastecimiento y desea registrar las cotizaciones recibidas. |
| **Flujo Principal**         | 1. El usuario accede a la vista de  **Cotizaciones**.<br>2. Selecciona el menú **Registrar Cotizaciones Recibidas**.<br>3. El usuario visualiza lista de solicitudes de abastecimiento que requieren cotizaciones.<br>4. El usuario selecciona la solicitud correspondiente.<br>5. Ingresa la información de la cotización enviada por el proveedor.<br>6. Añade una cotización por cada proveedor.<br>7. Confirma las cotizaciones para que sean ingresadas.<br>8. El sistema vincula las cotizaciones a la solicitud y las marca como listas para evaluación. |
| **Postcondiciones**         | Cotizaciones disponibles para evaluación. |
| **Excepciones**             | 1. "Cotización existente" Cuando la cotización para una misma solicitud ya se encuentra registrada.<br> 2. "Datos no Validos" Cuando se introducen datos con un valor no aceptado.<br>|
| **Frecuencia de Uso**       | Frecuente, cada vez que se reciben respuestas de proveedores. |

---

## **Caso de uso #7: Evaluar Cotizaciones y Generar Orden de Compra**

| **ID**                     | R-407 |
|-----------------------------|-------|
| **Actor(es)**               | Jefe de Abastecimiento |
| **Objetivo**                | Permite comparar las cotizaciones registradas y seleccionar la más conveniente en función de precio, tiempos de entrega y condiciones, generando la orden de compra correspondiente. |
| **Precondiciones**          |Cotizaciones registradas y asociadas a solicitudes de abastecimiento (R-406).|
| **Disparador o evento inicial**              | El usuario ingresa al módulo de Abastecimiento y desea evaluar las Cotizaciones. |
| **Flujo Principal**         |1. El usuario accede a la vista de Cotizaciones.<br>2. Selecciona el menú **Evaluar Cotizaciones Registradas**<br> 3. Selecciona la solicitud de abastecimiento que desea evaluar.<br>4. Visualiza y compara las cotizaciones registradas.<br> 5. Selecciona el proveedor que ofrece la mejor opción.<br>5. Genera la orden de compra al proveedor asociado. <br> 6. El sistema guarda la orden y notifica su creación.|
| **Postcondiciones**         | Existencia de una nueva entidad de Orden de Compra.  |
| **Frecuencia de Uso**       | Frecuente, cada vez que se realiza una compra a través de cotizaciones. |

---

## **Caso de uso #8: Monitorear y Reportar Compras**

| **ID**                     | R-408 |
|-----------------------------|-------|
| **Actor(es)**               | Jefe de Abastecimiento |
| **Objetivo**                | Permite monitorear el estado de las órdenes de compra emitidas y generar reportes periódicos. |
| **Precondiciones**          | Deben existir órdenes de compra en ejecución. |
| **Disparador o evento inicial**              | El usuario ingresa al módulo de Abastecimiento y desea monitorear una orden de compra. |
| **Flujo Principal**         | 1. El usuario accede a la vista de **Abastecimiento**.<br>2. Selecciona el menú **“Monitoreo de Compras”**.<br>3. Revisa el estado de las órdenes de compra en curso.<br>4. Si una orden se encuentra en estado **“Recepción”**, procede a controlarla.<br>5. Registra los lotes de productos recepcionados.<br>6. Registra la fecha y hora de recepción de la compra.<br>7. Actualiza el estado de la orden a **“Recibida”**. |
| **Flujo Alternativo I**     | 3. El sistema detecta que la orden no fue entregada en la fecha prevista.<br>4. Se genera el reporte de incumplimiento y cierra el monitoreo de esa orden.|
| **Flujo Alternativo II**    | 5. El usuario detecta que la entrega está incompleta.<br>6. Registra la diferencia de unidades recibidas.<br>7. El sistema actualiza la orden como **“Recepción Parcial”**.<br>8. El usuario completa el registro de fecha y hora de recepción.<br>9. Se genera el reporte con observación de recepción parcial.|
| **Flujo Alternativo III**   | 5. El usuario identifica productos dañados o en mal estado.<br>6. Marca el lote como **“Rechazado”** y adjunta evidencia (fotos, documentos).<br>7. El sistema actualiza la orden como **“Recepción Rechazada”** <br>8. Se genera el reporte con la observación de recepción rechazada.|
| **Postcondiciones**         | Reporte generado y compra disponible para descarga. |
| **Frecuencia de Uso**       | Periódico: semanal o mensual, aunque puede ser más frecuente en temporadas de alta rotación de inventario.|

## **Caso de uso #9: Gestionar Reclamos por Recepción**

| **ID**                     | R-409 |
|-----------------------------|-------|
| **Actor(es)**               | Jefe de Abastecimiento |
| **Objetivo**                |  Gestionar reclamos a proveedores por lotes incompletos, defectuosos o rechazados en la recepción. |
| **Precondiciones**          | Debe existir un reporte en la compra a la hora de su recepción. |
| **Disparador o evento inicial**              |  El usuario ingresa al módulo de Abastecimiento y desea gestionar un reclamo por recepción.|
| **Flujo Principal**         | 1. El usuario accede a la vista de Abastecimiento<br>2. Selecciona el menú Gestionar Reclamos por Recepción.<br>3. Visualiza la lista de órdenes con incidencias registradas.<br>4. Selecciona el reporte de recepción a reclamar.<br>5. Ingresa el motivo del reclamo y adjunta evidencia (documentos, fotos).<br>6. Envía el reclamo al proveedor.|
| **Excepciones**             |1. “Falta de evidencia” Cuando no se adjuntan archivos requeridos.|
| **Frecuencia de Uso**       | Poco frecuente, solo cuando ocurren incidencias en las recepciones.|

# **Requisitos de Atributos de Calidad**

| Requerimiento | Atributo de Calidad | Razón de Necesidad | Expectativa |
|---------------|---------------------|--------------------|-------------|
| **R-401 – Registrar Proveedor** | Usabilidad | El jefe de abastecimiento debe poder registrar proveedores de forma rápida y sencilla. | La interfaz debe permitir registrar un proveedor en máximo **3 pasos** y validar datos obligatorios como RUC y contacto. |
| **R-402 – Actualizar Proveedor** | Usabilidad | El sistema debe permitir actualizar datos sin complejidad para evitar errores. | La edición debe completarse en menos de **2 minutos** y con validaciones automáticas de RUC duplicado. |
| **R-403 – Registrar Producto Nuevo** | Usabilidad | Agregar productos debe ser ágil para mantener actualizado el catálogo. | El registro de un producto no debe requerir más de **6 campos obligatorios** y debe ser intuitivo. |
| **R-404 – Atender Solicitud de Abastecimiento** | Rendimiento | El jefe debe poder aceptar/rechazar solicitudes sin demoras. | El sistema debe mostrar solicitudes pendientes en menos de **2 segundos**. |
| **R-405 – Gestionar Solicitud de Abastecimiento** | Rendimiento | Es necesario priorizar solicitudes con rapidez para no retrasar compras. | La vista de solicitudes debe cargar en menos de **3 segundos** con hasta 500 registros. |
| **R-406 – Registrar Cotizaciones Recibidas** | Seguridad | Las cotizaciones son documentos formales que deben protegerse. | Solo usuarios autenticados podrán registrar cotizaciones, y estas quedarán  en un historial. |
| **R-407 – Evaluar Cotizaciones y Generar Orden de Compra** | Seguridad | Las órdenes de compra implican compromisos económicos. | Solo el jefe de abastecimiento podrá generar órdenes de compra, registrando fecha y usuario responsable. |
| **R-408 – Monitorear y Reportar Compras** | Rendimiento | El sistema debe permitir generar reportes de compras sin demoras. | Los reportes de hasta **1,000 registros** deben generarse en menos de **3 segundos**. |
| **R-409 – Gestionar Reclamos por Recepción** | Seguridad | Los reclamos deben estar respaldados con evidencia y sin riesgo de pérdida. | Los archivos adjuntos (fotos o PDF) deben guardarse correctamente y ser recuperables en cualquier momento. |
