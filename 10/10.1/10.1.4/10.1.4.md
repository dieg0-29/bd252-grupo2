# 10.1.4. Módulo 4 / Integrante 4

# Indices 🗃️
## Se da prioridad a tablas que presentaran gran cantidad de registros y que seran consultadas con regularidad

### DETALLE_PEDIDO por estado
**Importancia: CRÍTICA.** Este índice es vital para la pantalla I-012 (R-405: Priorizar Pedidos). Esta pantalla debe filtrar la tabla `DETALLE_PEDIDO` (que contendrá miles de items) para encontrar solo aquellos con `estado = 'Revisado'`. Un escaneo secuencial (Seq Scan) en esta tabla por el campo `estado` sería la operación más costosa de todo el módulo.

```sql
--SIN INDICE
EXPLAIN ANALYZE
SELECT cod_pedido, cod_producto, cantidad_requerida, fecha_requerida
FROM MODULO_ABASTECIMIENTO.DETALLE_PEDIDO
WHERE estado = 'Revisado';

--CREAMOS INDICE
CREATE INDEX idx_detalle_pedido_estado
ON MODULO_ABASTECIMIENTO.DETALLE_PEDIDO(estado);

--CON INDICE
EXPLAIN ANALYZE
SELECT cod_pedido, cod_producto, cantidad_requerida, fecha_requerida
FROM MODULO_ABASTECIMIENTO.DETALLE_PEDIDO
WHERE estado = 'Revisado';
```

### PROVEEDOR por nombre_comercial
**Importancia: ALTA.** Esta es la principal forma de búsqueda de proveedores en el sistema. Se usa en las pantallas I-001 (Listar Proveedores), I-005 (Editar Proveedor) y, de forma crucial, en la I-014 (R-406: Registrar Cotización) para seleccionar al proveedor que envía la oferta. Se usa varchar_pattern_ops para optimizar búsquedas con ILIKE 'texto%'.

```sql
--SIN INDICE
EXPLAIN ANALYZE
SELECT cod_proveedor, razon_social, ruc
FROM MODULO_ABASTECIMIENTO.PROVEEDOR
WHERE nombre_comercial ILIKE 'ACER%';

--CREAMOS INDICE
CREATE INDEX idx_proveedor_nombre_comercial
ON MODULO_ABASTECIMIENTO.PROVEEDOR(nombre_comercial varchar_pattern_ops);

--CON INDICE
EXPLAIN ANALYZE
SELECT cod_proveedor, razon_social, ruc
FROM MODULO_ABASTECIMIENTO.PROVEEDOR
WHERE nombre_comercial ILIKE 'ACER%';
```

### PRODUCTO_PROVEEDOR por cod_proveedor
**Importancia: ALTA.** Este índice es la respuesta directa a "buscar todos los productos que ofrece un proveedor". Es esencial para la pantalla I-005 (R-402: Actualizar Proveedor). Cuando un usuario edita un proveedor, el sistema debe cargar su catálogo. La llave primaria de esta tabla es (cod_producto, cod_proveedor), la cual no puede ser usada eficientemente para buscar solo por cod_proveedor. Este índice es la solución.

```sql
--SIN INDICE
EXPLAIN ANALYZE
SELECT p.nombre_producto, pp.precio_unitario_ref
FROM MODULO_ABASTECIMIENTO.PRODUCTO_PROVEEDOR pp
JOIN MODULO_ABASTECIMIENTO.PRODUCTO p ON pp.cod_producto = p.cod_producto
WHERE pp.cod_proveedor = ?;

--CREAMOS INDICE
CREATE INDEX idx_prodprov_cod_proveedor
ON MODULO_ABASTECIMIENTO.PRODUCTO_PROVEEDOR(cod_proveedor);

--CON INDICE
EXPLAIN ANALYZE
SELECT p.nombre_producto, pp.precio_unitario_ref
FROM MODULO_ABASTECIMIENTO.PRODUCTO_PROVEEDOR pp
JOIN MODULO_ABASTECIMIENTO.PRODUCTO p ON pp.cod_producto = p.cod_producto
WHERE pp.cod_proveedor = ?;
```
